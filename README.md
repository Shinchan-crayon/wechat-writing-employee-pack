# 公众号写作员工包

这是一个给公众号创作者用的 AI 写作工作流包。

它把一篇公众号文章从“选题素材”到“写完审校”拆成几个清楚的步骤，让 AI 不再一上来就乱写，而是先找素材、核事实、搭框架、写正文、做审校，最后再交给你确认。

简单说：你给它一个选题、素材、链接或想法，它会按流程帮你产出一篇更可靠、更像人写的公众号文章。

## 它能解决什么问题

很多人用 AI 写公众号，会遇到这些问题：

- AI 一上来就写，结构看着顺，但事实不一定准
- 文章里有数据、价格、融资、排名，却没有来源
- 标题像模板，正文也有明显 AI 味
- 每次都要重复提醒 AI：先查资料、别瞎编、按我的风格写
- 写完还要自己检查日期、排版、事实、风险表述
- 想接入公众号草稿箱，但又担心密钥、权限和误发布

这个员工包就是把这些重复要求变成固定流程。它不会让一个 AI 角色从头包到尾，而是让不同“员工”各做一段：该找资料的找资料，该核事实的核事实，该写的写，该审的审。

## 适合谁

适合：

- 想用 AI 辅助写公众号，但不想要一眼 AI 味的人
- 写科技、财经、商业、行业分析类文章的人
- 经常需要核对数据、价格、案例、引用的人
- 想把自己的写作风格沉淀成固定流程的人
- 想先生成可复制到公众号编辑器的 HTML 稿件的人

不适合：

- 只想要一句话生成全文，不关心事实和质量
- 主要写纯情绪、纯故事、纯娱乐八卦内容
- 希望 AI 自动最终发布文章，不做人工确认
- 没有任何素材，也不愿意让 AI 先做资料整理

## 最终会产出什么

默认产出是一篇公众号 HTML 稿件，可以复制到公众号编辑器里继续调整。

整个流程还会产出：

- 文章框架
- 事实核查表
- 标题候选
- 审校记录
- 可归档的最终稿

如果你配置了发布员，它可以把归档后的 HTML 推到公众号草稿箱。发布员只创建草稿，不会替你最终群发。

## 协作流程

```
账号配置员 ──→ 素材捕手 ──→ 事实核查员 ──→ 写手 ──→ 审校 ──→ 发布员（可选）
 (账号+配置)    (搜+搭)      (核数据)      (写+标)    (降+检+存)   (推草稿)
```

每个员工有硬门禁：上一个不交付，下一个不启动。

## 每个员工分别做什么

| 员工 | 你可以理解成 | 它负责什么 | 是否必需 |
|------|-------------|------------|:---:|
| 账号配置员 | 入职配置的人 | 确认账号、工作区、写作风格、用户本地配置 | 是 |
| 素材捕手 | 资料助理 | 读素材、补搜索、整理来源、搭文章框架 | 是 |
| 事实核查员 | 审事实的人 | 检查数据、价格、案例、引用，写前写后都核查 | 是 |
| 写手 | 正文作者 | 按核查通过的框架写 HTML 正文和标题 | 是 |
| 审校 | 主编 | 降 AI 味、查日期、做自检、归档 | 是 |
| 发布员 | 发草稿的人 | 把归档 HTML 推到公众号草稿箱 | 可选 |

第一次使用时必须先启动「账号配置员」。它会帮你生成用户本地配置，确认公众号账号、工作区和风格卡。如果你需要草稿推送，它再引导你配置 AppID、AppSecret 和网络允许列表。

## 快速开始

推荐安装完整总包：

```bash
npx skills add https://github.com/Shinchan-crayon/wechat-writing-employee-pack
```

安装后，对你的 AI 助手说：

```text
请读取公众号写作员工包，先执行账号配置员。不要直接跳到写手。
```

然后按提示完成三件事：

1. 生成本地 `USER.md`
2. 填写你的公众号账号、工作区和写作风格
3. 选择是否启用发布员

如果你只是想先体验写作流程，可以先不要启用发布员。这样不需要配置 AppID、AppSecret，也不会碰公众号草稿箱。

## 第一次应该怎么跑

第一次交付给用户时，建议先阅读并执行：[首次运行自检与故障反馈.md](./首次运行自检与故障反馈.md)。

首次加载本包时，Agent 应按以下顺序处理：

1. 检查用户本地配置是否存在
2. 引导用户基于 `用户配置模板/README.md` 生成 `USER.md`
3. 填写公众号账号、工作区、风格卡
4. 询问是否启用发布员
5. 如启用发布员，引导用户获取 AppID、AppSecret，查询本机出口 IP，并完成平台网络允许列表配置
6. 配置完成或用户选择跳过发布后，进入素材捕手

## 写作风格怎么控制

`共享资源/style.md` 提供了一份参考风格（适配科技/财经类公众号，说话犀利、短句、有网感）。

**这不是唯一风格。** 用户可以：
1. 直接用这份风格
2. 复制一份改成自己的 → 在 manifest.yaml 里指向新文件
3. 按 `共享资源/style-schema.md` 的字段定义写一份全新的

## 隐私和安全

这个包不会把你的用户配置放进公开仓库。

- `USER.md` 只在你本地生成
- `.env`、真实 AppSecret、真实 access_token 不进入公开包
- 发布员默认关闭
- 发布员只创建草稿，不做最终发布
- 没有配置平台凭证时，只运行写作闭环

## 给开发者看的设计说明

本员工包内部遵循以下设计规范：

1. **IDENTITY.md / SOUL.md 只负责软约束** — 定义员工是谁、怎么说话、有什么价值观、偏好什么输出风格。不承担安全边界
2. **policy.yaml 负责硬边界** — 工具白名单、文件访问范围、网络访问、脚本执行权限。由 adapter 编译成各运行时真实支持的权限配置
3. **USER.md 不进入公开包** — 用户画像、工作空间信息、私密偏好、密钥在安装后生成用户本地配置。模板见 `用户配置模板/`
4. **Skill 按最强 runtime 设计** — canonical 格式对齐 Claude Code 的 SKILL.md 思路，再按 `adapters/` 配置向 Hermes / Codex 做能力协商
5. **Adapter 是能力协商器** — 不是格式转换器。`adapters/` 目录下三个 yaml 声明各 runtime 的能力矩阵和降级策略

## 兼容性

| Runtime | 状态 | 说明 |
|---------|------|------|
| Claude Code | ✅ 全功能 | 全量加载 identity/soul/policy/skills |
| Hermes | ✅ 目标运行时 | 支持完整写作流程；浏览器搜索和发布引导依赖本地浏览器连接、登录状态和网络可用性 |
| Codex | ⚠️ 部分降级 | browser 受限，MCP 不可用，gates 改为用户提示 |

## 能力边界

- 面向中文科技/财经/行业分析类公众号
- 输出格式：HTML 文件（可直接粘贴到公众号编辑器）
- 默认无配图，用户要求时才生成
- 可选发布功能由账号配置员在首次运行时引导配置，只推送到草稿箱，不替用户最终发布
