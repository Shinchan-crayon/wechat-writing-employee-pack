# 踩坑大全

从实战中沉淀的高频失败模式和纠正方法。所有员工在各自阶段执行前必须过一遍。

---

## 素材编辑常见坑

### 坑1：没理解叙事就提炼角度
**症状：** 源素材涉及不熟悉的领域，凭表面信息自行提炼角度，写出跟素材完全无关的版本。
**纠正：** 动笔前先承认「这个领域我不熟」，向用户确认叙事方向和数据要点。问自己——用户给的素材里，哪句话是他说了但你没写的？如果有，说明理解不到位。

### 坑2：把搜索素材当叙事主轴
**症状：** 用户给了特定源素材要求写稿，但搜索到更劲爆的内容后就替换了源素材的核心角度。
**纠正：** 补充搜索仅用于验证数字/事实、补充背景信息。核心叙事和角度必须来自用户提供的源素材。自测：删掉所有补充搜索的内容后，文章还能成立吗？

### 坑3：多源素材默认每链接=独立任务
**症状：** 用户连续发多个链接，误以为每个链接要写一篇独立文章。
**纠正：** 先确认意图——是替换上一篇，还是融合成一篇，还是额外新任务。同主题多链接默认融合。

### 坑4：不搜索就写
**症状：** 只依赖单篇原文素材，不搜索补充来源。
**纠正：** 每次至少搜 3-5 篇不同来源的报道。搜索完成后必须把搜集到的素材链接发给用户确认。

### 坑5：X 帖子人物姓名写进正文
**症状：** 正文中出现源素材 X/Twitter 帖子里的个人博主姓名。
**纠正：** 用「有人」「社区有人」「据链上数据显示」等匿名指代。官方媒体（如路透、彭博）名称可保留。自测：通读全文，搜「在 X 上」「发帖说」「监测到」，每命中一处判断是官方媒体还是个人博主。

## 数据核查常见坑

### 坑6：Token 定价体系混淆
**症状：** 不同平台的「token」含义完全不同（大模型计费单位 vs 产品积分），混用导致计算全错。
**纠正：** 确认具体平台 → 搜索该平台定价 → 多方校准 → 标注信源。不输出未经核验的推算结论。

### 坑7：把投行预测当事实
**症状：** 券商/投行的预测数据被当作「已确认的事实」写进正文。
**纠正：** 区分「已确认」（公司财报/官方公告）和「预测」（分析师预期）。预测必须标注信源和前提条件。

### 坑8：二手材料当主口径
**症状：** 关键数字只用二手分析来源，未找到官方/一手来源。
**纠正：** 数据来源优先级：官方财报/公告 > 权威媒体 > 二手分析 > 社交媒体。关键数字只能用前两级。

### 坑9：换算不跑工具
**症状：** 汇率换算、每天/每秒拆分靠心算，结果出错。
**纠正：** 所有换算用 Python/命令行算，不心算。

## 写作常见坑

### 坑10：写完只在对话框扔文字，不出 HTML
**症状：** 正文写完只在对话里展示，没有生成 HTML 文件存盘。
**纠正：** 写稿不是对话。交付的是 HTML 文件路径，不是对话框里的文字。自检：HTML 文件存在吗？路径对吗？是用模板改的吗？

### 坑11：从零手写 HTML
**症状：** 凭记忆手写 HTML，缺少 DOCTYPE、CSS、胶囊样式。
**纠正：** 始终从模板复制骨架，只替换标题/摘要/正文。不动文件名、不动类名、不动标签结构。

### 坑12：胶囊 CSS 丢失
**症状：** 多轮重写 HTML 后，`.titlestyle span` 的 `padding:6px 24px; border-radius:50px;` 丢失，红色胶囊不可见。
**纠正：** 交付前必须确认这两项 CSS 属性存在。

### 坑13：标题太虚
**症状：** 标题用了「变了」「快看」「来了」等模糊动词但没有具体信息量。
**纠正：** 删掉标题里所有虚词，剩下的字还能独立成立吗？不能 → 重写。

### 坑14：编造叙事框架
**症状：** 源素材有明确的叙事框架（如「复盘最强叙事」），写手自己发明概念替代（如改成「叙事寄生」）。
**纠正：** 必须使用源素材的框架，不得自己发明概念。源素材中有具体公司名、产品名、数据点必须引用进正文，不得用虚词替代。

### 坑15：不想象读者的陌生感
**症状：** 文章主角是读者不熟悉的公司/人物，但开头直接扔事件，没做简介。
**纠正：** 判断标准：普通读者听过这个名字吗？没听过 → 在事件第一句之后、后续展开之前加 1-2 句简介（成立时间、主营业务、规模）。

### 坑16：举例反复用「写文章」
**症状：** 每逢举例就用写文章、写标题、写小红书选题。
**纠正：** 用生活场景举例——旅游规划、搬家、看病挂号、装修、做饭。自测：Ctrl+F 搜「写文章」「选题」「标题」，超过 2 处立即替换。

## 降AI 常见坑

### 坑17：只换词不拆结构
**症状：** 连续两轮降 AI 只做词汇层替换，但文章骨架（背景→事件→原因→影响→结论）纹丝不动，朱雀仍 100% AI。
**纠正：** 按优先级：结构层 > 句式层 > 词汇层 > 节奏层。架构问题不解决别碰词汇。

### 坑18：段落密度过高
**症状：** 一段塞 2-3 句，每段平均超 30 字。
**纠正：** 一句话一段。这是朱雀检测最关键信号，优先于词汇层。段落密度不达标时，词汇层所有操作无效。

### 坑19：灌口语标记灌爆文章
**症状：** 在已经有好节奏的原文上疯狂加「你想想」「讲真」「说真的」，结果比原版更僵。
**纠正：** 口语标记全文 ≤2 个，只在自然停顿处放，不硬塞。原文节奏好的地方只做词级替换。

### 坑20：降AI 时破坏 HTML 格式
**症状：** 降 AI 改稿时删除胶囊小标题、改变图片顺序、打乱节划分。
**纠正：** 降 AI 在词层面工作，不在结构层面工作。格式保护检查：胶囊小标题还在吗？图片位置变了吗？节划分改了吗？

### 坑21：降AI 后忘记恢复 CSS
**症状：** 降 AI 重写 HTML 时 CSS 从记忆手写，漏掉 padding 和 border-radius。
**纠正：** 降 AI 重写 HTML 时，CSS 必须从 template 复制，不凭记忆手写。

## 审校常见坑

### 坑22：归档日期写错
**症状：** 凭记忆写日期，同一天写错两次。
**纠正：** 归档前必须跑 `date +%-m.%-d` 确认当天日期。

### 坑23：跳过 verify 直接归档
**症状：** 文章写完，verify 没跑或没通过就归档了。
**纠正：** verify 脚本不通过 = 不准归档。宁可多修十轮也不交未过验证的稿。

### 坑24：跳过自检说「搞定了」
**症状：** 写完初稿就说搞定了，实际跳了数据核查、事实核查、降 AI、自检。
**纠正：** 交付前必须 10 步逐项核对，缺一不可。说「搞定了」之前先跑一遍自检表。

### 坑25：换方法前不解释
**症状：** 遇到障碍直接换方法，用户不知道为什么。
**纠正：** 换方法前必须先解释：①为什么当前不行、②新方法是什么、③效果和后果。

## 通用纪律

1. **不编造任何事实。** 找不到合适的真实案例 → 搜索 → 搜不到 → 删掉该段，用笼统表述替代
2. **不凭记忆写事实细节。** 每个关键事实点必须回到原文链接核实
3. **不跳过用户确认。** 搜索素材后、标题选定后、降 AI 前，都必须停等用户确认
4. **不替读者做总结。** 删除「这意味着」「这说明了」「可以看出」——直接陈述事实
5. **不把预测当事实。** 投行预测、KOL 推测 ≠ 已验证数据
