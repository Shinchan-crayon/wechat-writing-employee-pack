# 文章事实核查系统（强制 — 交付前最后一步）

> **用途：** 文章写完、交付前，最后一道**事实反手核查**。不核查不得交付。
> **来源：** 2026-05-19 用户要求建立系统化的交付前事实自查机制，杜绝「写时没注意、写完没查、用户发现」的翻车循环。
> **核心原则：** 写的时候靠规则预防，写完之后靠核查兜底。两道关卡，缺一不可。

---

## 前置认知

### 🚨 头号禁令：不得编造任何事实

不管文章多需要引子、多需要一个具体案例来带动情绪，**绝不允许编造：**
- 不存在的安全事件
- 不存在的名人名言
- 不存在的产品功能
- 不存在的公司声明
- 不存在的机构认证

**宁可删掉整段引子让文章变得平淡，也不准编一个案例来凑。** 找不到合适的真实案例 → 去搜索找 → 找不到 → 删掉那一段，用笼统的表述代替，或者换一个有真实来源的案例。

> 2026-05-19 实战教训：写「闲置旧手机做冷钱包」时，为了制造紧迫感，凭空编造了「Cobo 钱包插件刚被黑，安全公司慢雾证实是官方的 CDN 包被篡改」。该事件完全不存在，慢雾从未发布过该声明。被 Cobo 找上门，用户严肃批评。此为最恶劣的错误，零容忍。

### 两种错误类型：编造 vs 混淆

本系统核查以下两类事实错误，它们**成因不同，防范方法也不同**：

| 错误类型 | 定义 | 例子（本会话中的教训） | 防范方法 |
|---------|------|---------------------|---------|
| **编造 (Fabrication)** | 写了一个不存在的事件/数据/声明 | 编造「Cobo钱包被黑、慢雾证实」——该事件从未发生 | 找不到真实案例就删段，不编 |
| **混淆 (Misattribution)** | 把真实数据/设施/事件安到了错误的实体头上 | 把 xAI 的 Colossus 超算写成 Anthropic 的 | 联合关键词搜索「公司名+设施名」验证归属 |

**区别处理：**
- 编造：零容忍，核心禁令。发现即删除整段，不准用任何理由保留
- 混淆：需要修复归属（改到正确的公司/实体名下），不能直接删除整段

### 为什么需要这道关卡

过去翻车记录（必须记住，以免再犯）：

| 事故 | 怎么犯的 | 教训 |
|------|---------|------|
| 20亿token=100块（示例平台A） | 直接抄原推文数据，没去官网验证定价平台 | 看到「token」先问「哪个平台的token」 |
| 券商预测当事实写 | 麦格理预测2027人均12.9亿→直接当"2027年每人发12.9亿"写 | 未公布业绩不能替人家算 |
| 推文内容当事件定论 | 用户单条推文的数据未交叉验证就写入正文 | 社交媒体是引子，不是数据来源 |
| **编造Cobo被黑事件（2026-05-19）** | 为了引子效果编造安全事件和慢雾声明 | 零容忍——找不到真实案例就删段，不编 |
| **把xAI的Colossus写成Anthropic的（2026-05-19）** | 混淆两家公司 | 核实每个公司/机构名，不确定的不写 |
| **推文"1万人预购"当事实写（2026-05-25）** | 萌小译AI项圈文章写了"1万人已下单"，数据仅来自一条推文，未经新闻来源验证 | 推文数据必须交叉验证后才写入 |

### 什么时候核查

**文章写完 → 事实核查 → 通过 → 交付**
**文章写完 → 事实核查 → 未通过 → 修正 → 再核查 → 交付**

核查是独立的步骤，不并入写作流程。

---

## 第一步：事实提取

把正文中所有**可验证的声称**列出来。不靠脑补，逐句扫。

### 需要提取的事实类型

| 类型 | 例子 | 为什么容易翻车 |
|------|------|--------------|
| **数字** | 价格、金额、百分比、分数、排名、日期、时长 | 最容易被单来源误导 |
| **产品名** | 模型名、平台名、项目名、工具名 | 同名异物的坑（Token/积分） |
| **事件声称** | "X公司做了Y""Y发生在Z时间" | 可能是谣言/过时信息 |
| **因果声称** | "因为A所以B""A导致了B" | 可能是单方面叙事 |
| **对比声称** | "比X快Y倍""超过Z" | 对比基准可能不公平 |
| **引用/转述** | "XX说……""据XX报道" | 转述可能失真 |

---

## 第二步：核实策略

对每个提取的事实，按以下优先级确定核实方法：

### 优先级 1：官方一手来源

| 事实类型 | 核实方法 |
|---------|---------|
| AI模型排行榜 | 上官网/榜单 |
| 产品/平台定价 | 上官方定价页 |
| 公司财报/营收 | 上官方IR页面或权威财经 |
| 产品发布时间 | 上官方公告/博客 |
| API价格 | 上官方定价文档 |
| 融资/投资 | 上官方新闻稿或Crunchbase |

### 优先级 2：权威二次来源

| 事实类型 | 核实方法 |
|---------|---------|
| 行业数据 | 第一财经、36氪、路透、Bloomberg等 |
| 排行榜数据 | 官方榜单页面（如SWE-bench、LMSYS） |
| 公开报道 | 横向对比2-3家媒体报道 |
| 项目信息 | GitHub README / Crunchbase / CoinGecko |

### 优先级 3：社交媒体

**社交媒体的数据只能作引子，不能作数据来源。** 除非该数据有官方链接佐证。

| 平台 | 使用规则 |
|------|---------|
| X/Twitter | 可用作"有这事儿"，但具体数字必须到官网/新闻查 |
| 小红书/抖音 | 仅作场景引子，不引用其中的数据 |

**2026-05-25 实战特别补丁（推文销售/预购数据）：** 推文中提到的"已售X件/预购Y人/收入Z"等数字，在未被至少一家正式新闻媒体（新浪、36氪、路透、TechCrunch等）交叉确认前，**一律视为"消息来源声称"而非"可核实数据"**，不得作为决定性的"卖出/预购/收入"数字写入正文。正确做法：用"推出后引发关注/迅速走红/成为话题"等笼统表述替代具体数字。

> 典型翻车：用户推特说"已有1万人预购" → 直接写"1万人已下单" → 实际上该数字无任何第三方来源确认 → 用户追问"在哪卖的" → 尴尬。

---

## 第三步：执行核查

### 通用核查流程

```
1. 列出所有需要核查的事实 → 形成核查清单
2. 对每个事实，按优先级确定核实方法
3. 逐个执行搜索 → 记录结果
4. 标记：✅ 已核实 / ⚠️ 有差异 / ❌ 无法核实
5. 全部过完后，判定能否交付
```

### 常见事实的核实要点

| 常见声明 | 常见陷阱 | 核实方法 |
|---------|---------|---------|
| "XX获投X亿" | 可能是估值不是融资额 | 查TechCrunch/36氪原文确认 |
| "排名第一/领先" | 特定基准下的排名 | 查最近榜单 |
| "比YY快X倍" | 特定测试条件下 | 查原文基准条件 |
| "用户X万/下载X万" | 自报数据未审计 | 找SensorTower等第三方 |
| "XX已关闭/下线" | 地区性下架 | 确认范围和条件 |

---

## 第四步：判定输出

### 交付门禁规则

核查完成后，对每个事实标记状态：

```
✅ 绿色 — 已核实（官方来源或2+权威来源交叉验证）
⚠️ 黄色 — 部分核实（有一个可信来源，无反向证据）
❌ 红色 — 无法核实（找不到来源，或来源之间有矛盾）
```

**交付规则：**

| 状态 | 能否交付 | 操作 |
|------|---------|------|
| 全部 ✅ | ✅ 可交付 | 按正常流程 |
| 有 ⚠️ | ✅ 可交付，但必须标注 | 在回执中标注"⚠️ 以下数据待确认" |
| 有 ❌ | ❌ 禁止交付 | 删除该数据或替换为可核实的数据，重新核查 |
| 核心数字 ❌ | ❌ 禁止交付 | 修正后再走完整流程 |

### 核查回执格式

交付时，在回执中附上核查结果（简短版）：

```
📋 事实核查结果
✅ 核心数字（SWE-bench 80.6%、融资3.45亿）→ 官方来源已核实
⚠️ 对比数据（"比Claude快3倍"）→ 仅找到单来源测试报告
✅ 人名/产品名 → 全部交叉验证
状态：全员通过，可交付
```

## 第五步：典型案例核查流程

### 案例D：AI公司基础设施归属核查（2026-05-19 新增）

```
声称："Anthropic租了22万张Nvidia显卡，Colossus 1超算"
❌ Colossus 1 是 xAI（马斯克）的，不是 Anthropic 的
核查：公司名+设施名联合搜索
规则：当文章提到硬件基础设施时，单独查所有权归属
```

### 案例F：人物/公司叙事/人设声称核查（2026-06-02 新增）

```
声称："塞勒「永不卖币」的人设破裂"
核查动作：
1. 搜索"Saylor + never sell + bitcoin + quote" — 只有媒体评论文章，没有塞勒本人说"我永不卖"的原始采访链接
2. 确认：该说法是币圈广泛传播的"共识性认知"，但没有可直接引用的原文链接
3. 结论：⚠️ 不可作为"人设破裂"式确定陈述写入，应改为事实描述（"四年首次净减持"）
4. 修正：用"四年没卖过币"替代"立人设/破人设"类叙事判断
```

**防翻车规则：**
- 人物/公司的"立场""人设""承诺""叙事"类声称（如"某人永不卖币""某公司承诺XX"），必须有原始出处链接（采访/推文/公告），不能仅凭行业共识
- 用户会对这类说法追问"原文在哪"——拿不出原文链接就是翻车
- 正确的做法：用事实行为替代叙事判断（"四年没卖过"—可验证的事实；"永不卖币"—不可验证的叙事）

### 案例E：推文销售/预购数据核查（2026-05-25 新增）

```
声称："萌小译AI宠物翻译器已有一万人预购"
核查动作：
1. 搜索"萌小译+预购+数量" → 无第三方来源
2. 搜索所有相关报道 → 只说"走红网络""引发关注"
3. 确认：该数字仅来源自一条X/Twitter推文
4. 结论：❌ 不可作为事实写入
5. 修正：改用"推出后迅速引发全网热议"
```

**防翻车规则：**
- 推文数据（尤其是销售/预购/收入数字）在新闻媒体交叉验证前，一律视为"消息来源声称"
- 如果没有任何一家新闻媒体引用该数字，说明媒体认为它不可信或不可核实
- 用户追问"哪里来的数据"时，拿不出来源 = 翻车

### 案例G：跨来源因果拼接核查（2026-06-03 新增）

```

```
场景：写诺基亚（NOK）AI转型文章时，引用了两个独立来源：
- 来源A：诺基亚Q1光学网络业务涨20%（律动报道）
- 来源B：老黄跟Marvell CEO同台说"未来AI拼的不是算力，是连接"，MRVL当天暴涨32%（律动相关报道）

❌ 错误写法：「诺基亚的20%涨幅，就是同一张桌子上分到的蛋糕」
  问题：把两个独立事件强行焊成因果关系。MRVL暴涨是因为老黄点名Marvell，诺基亚涨20%是它自己Q1业绩。
  两者是同一个行业趋势下的平行事件，不是因果链。

✅ 正确写法：「同一个趋势正在被市场验证——老黄说连接＞算力，MRVL暴涨32%。诺基亚Q1光学网络+20%。两条新闻并排看，方向一致」
  关键：保留"方向一致"的观察，去掉"A导致了B"的虚假因果。
```

**防翻车规则：**
- 从多个独立来源引用数据时，先问自己：这些事件之间是因果关系，还是同趋势下的平行事件？
- 避免用「分到蛋糕」「跟着喝汤」「同一张桌子」等暗示因果承接的表述，除非有明确证据（如A宣布后B跟涨的交易数据）
- 正确表述：并排呈现两个独立事实 + 点出共同趋势 → 不做因果焊接
- 用户会追问「这跟X有什么关系？」——答不上来 = 在强行关联

### 案例H：社交媒体个人细节核查（2026-06-04 新增）

```
场景：写币安广场负责人滢滢时，引用X帖子原文「离婚带三娃的人有这个精力」
核查动作：
1. 该描述出自 @BTC_Alert_ 的单条X帖子，属于KOL对当事人的个人观察/评价
2. 搜索所有其他来源的报道 → 没有任何新闻或官方信息确认滢滢的家庭情况
3. 结论：❌ 单来源社交媒体个人细节，不可作为事实写入正文
4. 修正：删掉「离婚带三娃」，改为「有决策权，能直接跟CZ汇报」
```

**防翻车规则：**
- 个人的家庭状况、年龄、婚姻、健康等信息，除非当事人本人公开声明或有权威媒体报道，否则**禁止写入正文**
- 单条X帖子/小红书的个人描述 ≠ 可核实的事实
- 如果用户追问「哪来的消息？」，而你的回答只能是「一条X帖子说的」→ 说明该信息未通过核查，必须删除
- 正确的做法：只写可验证的职业信息（职务、决策权限、汇报关系等）

### 案例I：行业推算与案例数据区分核查（2026-06-04 新增）

```
场景：写AI Agent代搭建赚钱文章时，声称「一单300到1000」「维护费每月200」
核查动作：
1. 搜索真实案例 → 有「我用扣子搭智能体赚了20万」的文章标题（链接失效），有B站变现案例视频
2. 搜索具体报价市场 → 知乎/CSDN有多篇文章讨论报价在几百到上千不等
3. 确认：300-1000是多个信息来源交叉估算出的行业价格区间，不是某一具体案例的实收数字
4. 结论：⚠️ 行业估算，有论坛讨论支撑但无具体可验证的案例链接
5. 修正建议：如果用行业推算数字，前面加「据行业交流/实测反馈」等限定语，或在交付时标注「价格数据来自行业社群讨论，非单一案例」
```

**防翻车规则：**
- 某实战类账号文章中的赚钱数字（一单赚多少、月入多少）必须有至少一个可点开验证的案例来源
- 行业逻辑推算（工具免费+需求存在=理论可行）不可直接替换为具体报价数字当事实写
- 正确做法：有真实案例 → 写案例和数据；无真实案例 → 用「可达」「据行业反馈」「理论上」等限定语
- 用户会追问「这是有人真赚到了还是你编的」——如果只能回答「行业逻辑推导的」，说明语气过实
