# 版本管理规范

这个文件记录公众号写作员工包的公开版本管理方式，方便用户判断当前 Skill 的变化范围，也方便后续在 ThinkAI Skill 广场维护版本记录。

## 当前版本

当前公开版本：`v1.0.0`

对应安装地址：

```bash
npx skills add https://github.com/Shinchan-crayon/wechat-writing-employee-pack
```

## 版本号规则

本项目使用语义化版本：

```text
v主版本.次版本.修订版本
```

例如：

```text
v1.0.0
v1.1.0
v1.1.1
v2.0.0
```

## 什么时候升级修订版本

修订版本用于小范围修复，不改变工作流结构。

适合：

- 修正文档错别字
- 优化 README 表达
- 补充调用示例
- 修正资源文件链接
- 修复不影响流程设计的小问题

示例：

```text
v1.0.0 -> v1.0.1
```

## 什么时候升级次版本

次版本用于新增能力，但仍兼容原来的总包工作流。

适合：

- 新增参考资料
- 新增标题公式或审校规则
- 增加可选模板
- 优化某个员工的执行说明
- 增加新的适配器配置
- 改进发布员的配置引导，但不改变安全边界

示例：

```text
v1.0.0 -> v1.1.0
```

## 什么时候升级主版本

主版本用于重大变化，用户需要重新理解使用方式。

适合：

- 调整员工顺序
- 拆分或合并核心员工
- 改变默认输出格式
- 改变发布员的权限边界
- 改变 USER.md 的核心字段结构
- 引入不兼容旧版本的配置方式

示例：

```text
v1.0.0 -> v2.0.0
```

## ThinkAI 上架记录建议

在 ThinkAI Skill 广场更新版本时，建议记录以下内容：

```text
版本号：
发布时间：
变更摘要：
影响范围：
是否需要用户重新配置：
是否影响发布员：
```

示例：

```text
版本号：v1.0.0
发布时间：2026-07-10
变更摘要：首次上架总包，提供公众号文章从素材整理、事实核查、正文写作到审校归档的完整协作流程。
影响范围：完整总包
是否需要用户重新配置：首次安装需要执行账号配置员
是否影响发布员：发布员为可选能力，默认不启用
```

## 发布前检查清单

每次发布新版本前，至少检查：

- `README.md` 是否说明清楚这个包是干什么的
- `manifest.yaml` 版本号是否更新
- 6 个员工目录是否仍然存在
- `USER.md`、`.env`、真实密钥是否没有进入公开仓库
- 安装命令是否仍然可用
- 发布员说明是否明确只创建草稿，不最终发布
- ThinkAI 上架文案是否和 GitHub 公开文档分开维护
